# 6. AI 활용 과정

> 과제 안내사항: *"AI를 활용했다면, 과제 이해·문제 정의·아이디에이션·리서치 등 어떤 과정에서 어떻게 활용했는지 최대한 구체적으로 명시."*
> 이 과제는 **Claude(Opus 4.8) 기반 에이전트 워크플로우(Claude Code)** 로 수행했다. AI를 "답을 대신 써주는 도구"가 아니라 **검증된 보조 + 병렬 리서처 + 설계 파트너**로 썼고, 사람(지원자)은 **방향 설정·단계별 검토·최종 판단**을 맡았다.

---

## 6.1 원칙 — AI를 어떻게 신뢰했나

1. **시간민감 사실은 기억이 아니라 출처.** 모델 가격·버전·SOTA는 학습 cutoff 이후 바뀌므로, 전부 **웹 1차 소스로 검증하고 URL·날짜를 부착**했다. (그렇지 않으면 면접에서 무너진다.)
2. **저작과 검증을 분리.** 생성 패스와 검증 패스를 분리하고, 인용·수치는 빌드 전 **adversarial 교차검증**한다.
3. **사람이 게이트.** 요구사항 확정·포지셔닝·설계 방향·빌드 착수마다 사람 승인 게이트를 뒀다(자율 실행 ≠ 무단 진행).

---

## 6.2 단계별 활용

### ① 과제 이해 & 문제 정의 — Socratic Deep Interview
- 곧장 답을 쓰지 않고, **구조화된 인터뷰(6라운드)** 로 요구사항을 결정화했다. 모호도를 0~1로 점수화해 **100% → 13%** 로 낮춘 뒤에야 작업을 시작.
- **토폴로지 게이트**로 작업을 3개 독립 컴포넌트([1]/[2]/산출물)로 잠그고, **Contrarian 모드**("트렌디함이 신뢰감을 깎지 않나?")·**Simplifier 모드**("과설계 아닌가?")로 가정을 흔들어 검증.
- 산출: 합의된 스펙(포지셔닝=실용+트레이드오프 논증+비용감각/시니어 IC, 산출물=웹1차+PDF, 기술 stance=오픈소스 대량+상용 전략적+한국어 우선).

### ② 아이디에이션
- 포지셔닝과 **"깔때기(funnel)" 설계 철학**, 그리고 **"비용은 binding 제약이 아니다"** 라는 핵심 논점을 AI와의 대화로 도출. 회사 맥락(레브잇/올웨이즈, AI 쇼핑 에이전트)을 반영해 *분류·속성 → 검색·추천·에이전트 → GMV* 임팩트 서사를 구성.

### ③ 리서치 — 6개 병렬 에이전트 (웹 검증·인용)
동시에 6개 서브에이전트를 띄워 각자 다른 레인을 조사하고, 모든 사실에 출처를 달게 했다:
| 레인 | 핵심 산출 |
|---|---|
| 한국어 모델·임베딩·VLM | KURE-v1(MIT) · Qwen2.5-VL(Apache) · Solar Pro 2 + **라이선스 함정**(EXAONE NC 등) |
| 상용 API 가격(batch) | 공식 pricing 직접 fetch → 단가·100만건 추산 |
| GPU 경제성·throughput | self-host 500만 작업별 비용(임베딩 $5 … VLM $350) |
| 분류 SOTA·산업사례 | Amazon Dual-Expert·Naver e-CLIP·Mercari, 아키텍처 비교 |
| 속성 induction·추출·평가 | 택소노미 시드·멀티-LLM 앙상블·멀티에이전트 debate·LLM-judge 편향 |
| 회사 맥락 | 레브잇/올웨이즈 식별(미국 levit.ai와 구분) |
- 종합은 `research-synthesis.md` 한 장으로. (이 문서 설계의 모든 수치·선택의 근거)

### ④ 설계
- 리서치를 종합해 **Opus가 4개 모듈 + 비용모델 + 로드맵을 직접 작성**, 사람이 검토·조타. 각 결정에 "대안 + 선택 이유 + 정량 비용"을 붙여 면접 방어력을 확보.

### ⑤ 빌드
- 단일 마크다운 소스 → **웹(GitHub Pages, 한국어·모바일·인터랙티브)** + **PDF** 동시 빌드. 톤은 절제된 트렌디(Linear/Stripe docs 풍).

### ⑥ 검증
- 빌드 전 **인용·수치 adversarial 검증**(미래시점 arxiv ID 등 의심 항목 1차 소스 확인 → 검증 실패 시 제거하거나 "사례 기반 추정"으로 강도 하향).

---

## 6.3 투명성 — 무엇이 검증됐고 무엇이 추정인가

- **검증됨(1차 소스):** 모델 가격(공식 pricing 직접 fetch), GPU 단가(RunPod·Lambda), 라이선스, 주요 산업사례(Amazon·Naver·Mercari).
- **추정(가정 명시):** 모든 비용 수치는 *개략 추정*이다. throughput·이미지 토큰은 ±30~50% 오차 가능. 단 **결론(자릿수)** 은 견고.
- **검증 대상으로 남긴 것:** 일부 최신 논문 인용은 출판 시점·ID를 빌드 단계에서 교차검증(검증 실패 시 강도 하향). 설계 자체는 *검증된 표준 기법* 기반이라 특정 인용의 진위와 무관하게 성립.

---

## 6.4 메타 교훈

이 과제를 푼 방식 자체가 답의 일부다. **deep-interview로 요구사항을 수렴 → 병렬 에이전트로 리서치를 검증·확장 → Opus로 설계를 합성 → 단일 소스로 산출물을 빌드** 하는 흐름은, 그대로 **AI/Data Product Engineer가 실무에서 LLM 파이프라인을 만드는 방식**이다(요구 정의 → 검증된 리서치 → 깔때기 설계 → 평가 루프). 본문의 분류·속성 파이프라인이 *"싼 모델 대량 + 비싼 모델 소수 + 사람 환류"* 인 것과, 이 과제 수행이 *"병렬 AI 대량 + 사람 게이트 검증"* 인 것은 같은 원리다.
