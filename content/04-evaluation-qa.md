# 4. 속성 체계 평가 및 품질 관리

> **과제 [2-3]** — 2-1에서 구축한 속성 체계(Key-Value 리스트)가 얼마나 **정확하고 빠짐없이** 구성됐는지 평가하고 고도화하는 방법(인간 피드백·자동 메트릭 등).

---

## 4.0 무엇을 평가하는가 — 두 층위

품질은 두 곳에서 갈린다. 분리해서 측정한다.

1. **스키마 품질(2-1)** — Key 집합이 *빠짐없고(CE)*, *겹치지 않고(ME)*, *맞는가(타입·정의)*.
2. **추출 품질(2-2)** — 그 스키마로 뽑은 값이 *정확한가(P/R/F1)*, *정규화가 맞는가*.

이 둘은 연결돼 있다. 스키마가 빈약하면(CE 미달) 추출이 아무리 정확해도 중요한 정보를 놓친다. 그래서 **CE(포괄성) 평가가 가장 중요하고 가장 어렵다.**

---

## 4.1 스키마 품질 메트릭 (2-1)

| 메트릭 | 정의 | 측정 |
|---|---|---|
| **Coverage (CE)** | 구매 고려 속성 중 스키마가 포함한 비율 | 전문가 golden Key 셋 대비 recall |
| **중복률 (ME)** | 의미 중복 Key 쌍 비율 | Key 쌍 임베딩 유사도 > 임계 자동 검출 |
| **타입 정확도** | Key 타입(enum/수치/…) 적절성 | 샘플 전문가 검수 |
| **Value 후보 완전성** | enum 후보가 실제 값을 포괄하는 비율 | 추출된 값 중 "기타" 비율(낮을수록 좋음) |
| **Annotator Agreement** | 어노테이터 간 Key 일치 | Cohen's Kappa |

### 포괄성(빠짐없음)은 어떻게 측정하나 — 어려운 문제와 네 가지 우회

"빠진 속성이 없다"는 *모르는 것을 측정*해야 하는 열린 문제다. 단일 지표로 못 풀고, **여러 약한 신호를 교차**한다.

1. **Golden set recall** — 전문가가 정의한 카테고리별 "필수 속성 정답"을 스키마가 얼마나 덮는가. (직접적이나 golden 자체가 완전하지 않을 수 있음)
2. **검색 로그 gap 마이닝 (강력·실전적)** — 사용자가 검색·필터에서 쓰는데 *스키마에 없는* 속성형 질의를 채굴한다. 예: "무라벨 생수"가 많이 검색되는데 스키마에 "라벨여부"가 없으면 → **CE 갭이 실데이터로 드러난다.** 다운스트림(검색)이 곧 포괄성 평가자.
3. **Held-out attribute 테스트** — 알려진 Key 하나를 일부러 빼고 파이프라인이 재발견하는지 검사 → 발견 메커니즘의 민감도 측정.
4. **경쟁사 스키마 diff** — 네이버·쿠팡·Amazon 동일 카테고리 속성 합집합 대비 누락 점검.

> 핵심: **검색 로그 gap 마이닝**은 Levit/올웨이즈 같은 실서비스에서만 가능한 강점이다. "측정 못 하는 CE"를 "실사용자가 원하는데 못 채운 속성"이라는 **측정 가능한 신호**로 바꾼다.

---

## 4.2 추출 품질 메트릭 (2-2)

- **Precision / Recall / F1 (micro-averaged)** — golden 대비 값 정확도.
- **값 정규화 정확도** — 정규화 후 값이 golden과 exact/semantic match 비율.
- **"없음" 분류 정확도** — 미기재/해당없음/추출실패 3종 구분이 맞는가(데이터 신뢰 직결).
- **벤치마크 활용** — WDC-PAVE(정규화 포함), ImplicitAVE(암묵 속성), MAVE로 방법론 사전 검증.
- **신뢰도 보정(calibration)** — 모델 confidence와 실제 정확도가 일치하는가(ECE). 라우팅 임계값의 신뢰성 근거.

---

## 4.3 Golden Set — 평가의 기준점

- 카테고리별 대표 상품 **100~200개 × 전체 Key**를 **전문가 2인 이상**이 어노테이션, 불일치는 제3자 중재/다수결.
- 용도: (a) 자동 메트릭의 정답, (b) 모델·프롬프트 선택 기준, (c) LLM-judge 캘리브레이션.
- **차등 투자:** 상위 트래픽 카테고리(~1,000개, GMV 대부분)는 풍부한 golden, 롱테일은 경량 golden + 자동 게이트. (1억 예산의 사람 비용이 주로 여기 — §4.6)

---

## 4.4 Human-in-the-Loop 설계

**2단계 검증(모델 자동 QC → 전문가 검증):**
1. **모델 QC** — 자동 품질체크로 명백한 오류(형식 위반·환각·중복 Key) 필터.
2. **도메인 전문가 검증** — 카테고리별 전문가가 Key·값·정의 검증.

**샘플링 전략(비용 효율):**
- **Uncertainty sampling** — 저신뢰 케이스 우선 검토(가장 비용효율적).
- **층화 샘플링** — 카테고리·속성·값분포를 골고루 뽑아 사각지대 방지.
- **무작위 샘플** — 전체 품질 모니터링 baseline.

## 4.5 LLM-as-Judge — 대량 평가, 단 편향 방어

전문가만으로 1만 카테고리·500만 상품을 평가할 수 없다. LLM-judge로 **1차 스크리닝**하되, 알려진 편향을 방어한다.

| 편향 | 방어 |
|---|---|
| Verbosity(장문 선호) | rubric 명시, 길이 무관 채점 기준 |
| Position(순서 선호) | **swap consistency**: 순서 바꿔 2회, 불일치 시 불신뢰 |
| Self-enhancement(자기 선호) | **이질 멀티-judge 앙상블**(3종+ 다수결), 생성≠평가 모델 분리 |
| 전반 신뢰도 | **golden calibration**: judge를 인간 golden에 보정, 편차 추적 |

**운영 원칙:** LLM 평가는 대량 1차 필터로만. **애매하거나 위험·고가치 카테고리는 사람에게 넘긴다.** 자동 평가 결과는 사람 샘플로 주기적으로 다시 맞춘다.

**차등 품질 게이트(예시):**
| 카테고리 등급 | Coverage 게이트 | 추출 F1 게이트 | 검수 강도 |
|---|---|---|---|
| Head(상위 ~1K, GMV 대부분) | ≥ 0.95 | ≥ 0.90 | 전수 전문가 |
| Torso | ≥ 0.90 | ≥ 0.85 | 샘플 전문가 |
| Tail(롱테일) | ≥ 0.80 | ≥ 0.80 | 자동 + 경량 샘플 |

> 1만 카테고리를 동일 강도로 다루지 않는다. **임팩트(GMV·트래픽)에 비례해 사람·예산을 차등 투입**하는 것이 1억을 가장 잘 쓰는 법이다.

**[2-3] 비용 성격:** 평가·QA의 비용은 거의 전부 **사람(golden 구축·전문가 검수)** 이다. compute(자동 메트릭·LLM-judge)는 수백 달러. → 1억은 *상위 카테고리 golden 구축 + 전문가 검수 라운드*에 쓰는 게 최적.

## 4.7 리스크와 완화

| 리스크 | 완화책 |
|---|---|
| Golden set 자체 불완전 | 다인 어노테이션 + 검색로그 gap로 보강, 주기 갱신 |
| LLM-judge 편향 신뢰 | swap·멀티judge·rubric·golden 보정, 고위험은 사람 |
| CE를 영원히 못 잡음 | 검색로그 gap 상시 마이닝 → 실수요 기반 신규 Key 자동 후보화 |
| 평가-운영 괴리 | 다운스트림 지표(검색 충족률·필터 사용률)를 최종 평가축으로 |
| 롱테일 평가 사각 | stratified 샘플 + 자동 게이트 + 이상치 탐지 |

> **결론:** 평가는 "한 번 점수 매기기"가 아니라 **검색로그·검수·다운스트림을 신호로 스키마·추출을 계속 끌어올리는 루프**다. 특히 CE는 *측정 불가의 영역*을 *실사용자 신호*로 바꿔 공략하는 것이 이 설계의 차별점이다.
