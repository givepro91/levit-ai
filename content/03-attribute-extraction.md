# 3. 상품별 속성값 추출 파이프라인

> **과제 [2-2]** — 2-1에서 정의한 속성 체계를 바탕으로, 500만 상품 각각의 본문/이미지에서 **정확한 속성값(Value)** 을 추출.

---

## 3.0 무엇이 어려운가

[1]·[2-1]과 결이 다른 난점이 있다.

1. **스키마 조건부(schema-conditioned).** 추출할 Key가 상품마다 다르다 — 상품의 카테고리([1] 결과)가 어떤 스키마(2-1)를 적용할지 결정한다. "생수"엔 수원지를, "노트북"엔 CPU를 묻는다.
2. **500만 × 멀티모달.** 텍스트(상품명·상세)뿐 아니라 **이미지에만 있는 속성**(무라벨, 색상, 형태)이 있다. 그러나 이미지 추론(VLM)은 텍스트보다 비싸다 — 무차별 VLM은 비용을 깬다.
3. **암묵적 속성(implicit).** 본문에 "500ml"는 있어도 "용량 종류=소용량"은 추론해야 한다. 명시값 추출 + 추론 추출이 섞여 있다.
4. **정규화(normalization)가 진짜 가치.** "500미리", "0.5L", "500ML"가 모두 `500ml`로 통일돼야 검색·필터·추천이 작동한다. 추출보다 정규화가 다운스트림 가치를 좌우한다.
5. **"없음"의 3종 구분.** *원문에 없음* vs *카테고리에 해당없음* vs *추출 실패* 는 전혀 다르다. 뭉뚱그리면 데이터 신뢰가 깨진다.

### 왜 텍스트 우선 2-pass인가 (비용 트레이드오프)

- VLM은 텍스트 LLM보다 건당 ~7~70× 비싸다(§5). 500만 전수 VLM은 ~$350~수천이지만, **대다수 속성은 텍스트에 이미 있다.**
- 따라서 텍스트로 먼저 최대한 뽑고(거의 공짜), **이미지가 정말 필요한 잔여 필수 Key가 있는 상품에만** VLM을 건다 → VLM 호출량을 전수의 일부로 축소.
- evidence_span(근거 구간)을 함께 뽑아 **감사 가능성(auditability)** 과 신뢰도 판단 근거를 남긴다.

## 3.2 핵심 설계 포인트

**Structured Output 강제.** JSON Schema를 모델에 전달해 형식 위반·환각 Key를 차단. 출력 예:
```json
{
  "용량":   {"value": "500ml", "confidence": 0.96, "evidence": "500ml 대용량", "source": "text"},
  "라벨여부": {"value": "무라벨", "confidence": 0.88, "evidence": "(이미지)",      "source": "image"},
  "수원지":  {"value": null,     "confidence": null, "reason": "미기재",           "source": "none"}
}
```

**"없음" 3종 구분.** `reason ∈ {미기재(원문에 없음), 해당없음(카테고리 부적합), 추출실패(모델 불확실)}`. 이 구분이 (a) 데이터 신뢰, (b) 어디를 보강할지(미기재→추가 크롤/추론, 추출실패→재시도/사람) 결정한다.

**암묵적 속성.** 명시 안 된 속성은 (a) VLM(이미지 추론), (b) LLM 사전지식·RAG(예: 브랜드→원산지)로 보강하되, **추론값은 source="inferred"로 명확히 태깅**하고 신뢰도를 보수적으로. (명시값과 추론값을 섞지 않는다.)

**정규화 = 다운스트림 가치.** 단위 계산이 필요한 정규화는 LLM도 여전히 약한 영역이므로 **규칙 우선 + 복잡 케이스만 LLM**. 정규화한 값을 스키마의 표준 후보값에 맞춰, 검색·필터에 바로 쓰이는 구조값으로 만든다.

**어려운 건 여러 모델로 교차검증.** 확신이 낮거나 모순되거나 중요한 카테고리는 서로 다른 LLM 여러 개가 의견을 주고받아(debate) 합의로 견고함을 확보한다. 한 모델이 지어낸 값을 교차검증으로 걸러낸다.

## 3.3 [2-2] 비용 모델 (상세 §5)

> 500만 전수, 가정 명시. self-host = RunPod, 환율 1,350원/USD.

| 단계 | 대상 | 추정 비용 | 비고 |
|---|---|---|---|
| Pass 1 텍스트 추출 | 500만 전수 | ~$50~150 | self-host LLM, 1회 |
| Pass 2 VLM 추출 | 이미지 필요 부분집합(~30%, 150만) | ~$100~500 | self-host VLM, 부분집합 |
| 정규화 | 500만 | ~$10~50 | rule 위주 |
| 하드케이스 상용/debate | ~15%(75만) | ~$200~3,000 | 모델 등급에 따라 |
| 사람 검수 | 필수 Key 미해결 ~5%(25만) | ₩10M~30M | 진짜 비용 |
| **합계** | | **≈ ₩2,000만~5,000만** | **1억의 20~50%** |

→ 만약 *전수 VLM + 전수 상용 LLM* 으로 순진하게 돌리면 멀티 Key·멀티이미지로 **$50K~150K+** 까지 치솟아 1억을 위협한다. 2-pass 깔때기가 이를 막는 핵심 장치다. 남는 예산은 **상위 카테고리 사람 검수 확대 + 하드케이스 상용 모델 등급 상향**에 재투자.

## 3.4 정확도/효율 레버

| 레버 | 저비용 방향 | 고정확도 방향 |
|---|---|---|
| 모델 | self-host 7B fine-tune | 상용 GPT-4o/Claude/Gemini |
| 입력 | 텍스트만 | 텍스트 + 이미지(VLM) |
| Key 범위 | 필수 Key만 | 전체 스키마 |
| 암묵 속성 | 스킵 | VLM/RAG 추론 |
| 검증 | 단일 모델 | 멀티에이전트 debate |

- 가장 큰 정확도 레버는 **카테고리별 fine-tuning**(상위 카테고리에 특화 student)과 **few-shot 예시**(2-1 스키마의 예시·반례를 프롬프트에).
- 가장 큰 비용 레버는 **2-pass(텍스트→이미지 선택)** 와 **batch(50% 할인)**.

## 3.5 리스크와 완화

| 리스크 | 완화책 |
|---|---|
| 환각값(없는 값 생성) | structured output + evidence_span 필수, span 없으면 신뢰도 강등, 멀티에이전트 교차검증 |
| 정규화 오류(단위 계산) | 규칙 우선, 복잡 케이스만 LLM, 정규화 결과 별도 검증셋 |
| VLM 비용 폭증 | 2-pass(이미지 필요 시만), 부분집합 한정, batch |
| 이미지 누락·저품질 | 품질 필터, 텍스트 fallback, source 태깅 |
| 암묵/명시 혼동 | source(text/image/inferred) 태깅, 추론값 신뢰도 보수화 |
| 카테고리 오분류 전파([1] 오류) | [1] 신뢰도 낮은 상품은 속성추출도 보수적/사람 라우팅 |
| 스키마-상품 불일치(해당없음) | reason="해당없음" 명시, [2-3] 피드백으로 스키마 보정 |

> **연결:** 추출 품질의 측정(F1·정규화 정확도)과 2-1 스키마 품질(coverage·MECE)의 평가·개선은 4장(2-3)에서 통합해 다룬다.
