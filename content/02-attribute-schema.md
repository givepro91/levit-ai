# 2. 카테고리별 속성 스키마 구축 파이프라인

> **과제 [2-1]** — 1만 카테고리 각각에 대해, 구매에 필요한 **필수 속성(Key)** 과 들어갈 수 있는 **후보 값(Value) 집합** 을 정의.
> 제약: 카테고리별 독립 스키마, **MECE** (특히 **포괄성 CE** — 소비자가 구매 시 고려하는 모든 핵심 정보를 커버).

---

## 2.0 무엇이 어려운가 — 1만 개 스키마를 빠짐없이·겹치지 않게

예시(과제): `생수` 카테고리 → Key: 수원지(지리산), 용량(500ml), 라벨 여부(무라벨).

카테고리마다 사전을 만드는 단순 작업처럼 보이지만, 진짜 난점은 셋이다.

1. **규모 × 독립성.** 1만 개 leaf 각각이 고유 스키마를 갖는다. 1만 번을 따로 만들면 (a) 비용·시간 폭증, (b) 일관성 붕괴("용량" vs "용량(ml)" vs "내용량"이 카테고리마다 제각각).
2. **포괄성(CE) 증명의 어려움.** "빠진 속성이 없다"를 어떻게 보장하나? 생성은 쉽지만 *누락 없음*을 증명하는 건 본질적으로 어렵다(열린 문제). 단일 LLM은 자기 사각지대를 모른다.
3. **상호배타성(ME)·정규화.** "수원지"와 "원수원지", "무라벨"과 "라벨없음"이 다른 Key/Value로 난립하면 안 된다.

핵심 통찰: **1만 개를 백지에서 독립 생성하지 않는다.** 카테고리들은 속성을 대거 공유한다(모든 식품은 용량·유통기한, 모든 가전은 정격전압·크기). → **공유 속성 라이브러리(shared attribute library) + 카테고리별 오버레이** 구조로 가면 비용·일관성·포괄성을 동시에 잡는다.

### 왜 이 조합인가 (트레이드오프)

| 접근 | 장점 | 약점 | 우리 선택 |
|---|---|---|---|
| LLM 단독 생성 | cold-start 강함, 빠름 | 모델 bias·할루시네이션, 포괄성 미보장 | 보조로만 |
| 데이터마이닝 단독 | 현실 반영, 검증 가능 | 신규/롱테일 카테고리 데이터 부족, 판매자 누락 속성 못 잡음 | 뼈대로 |
| **혼합(마이닝 뼈대 + 멀티-LLM 포괄성 + critic)** | 현실성 + 포괄성 + 검증 가능성 | 파이프라인 복잡 | **채택** |

단일 LLM이 아니라 **서로 다른 LLM 여러 개**를 쓰는 이유: 한 모델의 사각지대를 다른 모델이 메운다. 포괄성(CE)은 "다양성"으로 근사하는 게 현재 최선이다(여러 모델이 독립 생성 → 집계, 그리고 별도 critic이 누락을 잡는 멀티-에이전트 스키마 구축 접근).

## 2.2 Value 후보 집합과 타입 체계

각 Key는 **타입**을 갖고, 타입에 따라 Value 후보를 다르게 관리한다.

| Key 타입 | 예시 | Value 관리 | 정규화 |
|---|---|---|---|
| **enum(범주)** | 라벨 여부 → {무라벨, 일반라벨} | 후보 집합 고정 + "기타" 허용 | 동의어 매핑(라벨없음→무라벨) |
| **수치+단위** | 용량 → 500ml | {min,max,unit} 또는 정규화 수치 | 단위 변환(L↔ml), 범위 파싱 |
| **entity(개체)** | 수원지 → 지리산 | 개방형 + 정규화 사전(지명) | 표기 통일(지리산/智異山) |
| **boolean** | 무선 여부 → true | {true,false} | "있음/지원"→true |
| **free-text** | (지양) | 최후수단 | 가능하면 enum화 |

> 설계 원칙: **가능한 한 enum/수치로 닫는다.** free-text는 검색·필터·추천에서 쓸모가 떨어지므로 최소화한다(이게 다운스트림 임팩트로 직결).

---

## 2.3 필수(essential) vs 선택(optional) — 구매 관련성 랭킹

과제는 "**필수** 속성"을 요구한다. 모든 속성이 동급이 아니다.

- **구매 관련성 점수(purchase-relevance)** 로 Key를 랭킹: "이 속성을 모르면 구매 결정을 못 하는가?"
- 신호: (a) LLM 판단(구매 가이드 기반), (b) 검색 로그(해당 속성으로 필터·검색하는 빈도), (c) PDP 노출 빈도(판매자가 항상 적는가).
- 결과: 카테고리별 Key를 **필수 / 권장 / 선택** 3등급으로. 생수라면 필수=용량·종류, 권장=수원지·라벨여부, 선택=경도.
- → 2-2 추출에서 **필수 Key부터 추출**해 비용을 우선순위화한다.

## 2.4 MECE 운영화 — CE와 ME를 따로 공략

**포괄성(CE, Collectively Exhaustive):**
1. 멀티-LLM 앙상블(다양성) + RAG(외부 지식) + 경쟁사 스키마 합집합(네이버·쿠팡·Amazon 동일 카테고리).
2. **Completeness critic** 전용 패스(빠진 것 찾기).
3. **소비자 구매여정 분석** 프롬프트("구매 전 비교 시 확인하는 것?").
4. 측정: 전문가 golden Key 셋 대비 **coverage(recall)** (→ 2-3).

**상호배타성(ME, Mutually Exclusive):**
1. 임베딩+편집거리 동의어 클러스터링 → Key 병합.
2. **계층 Key 구조**: 겹치는 개념을 부모-자식으로(연결 > Wi-Fi/Bluetooth) → 중복 대신 구조.
3. **Key 정의서**: 각 Key에 정의 + 예시 + 반례를 명문화(어노테이터·LLM 공용 가이드) → 경계 모호 제거.
4. 측정: Key 쌍 임베딩 유사도 > 임계 → 중복 의심 자동 검출(→ 2-3).

## 2.5 [2-1] 비용 — 여기선 compute가 아니라 "사람"이 비용

> 결정적 차이: 2-1은 **카테고리 단위(1만)** 작업이지 상품 단위(500만)가 아니다.

| 항목 | 추정 | 비고 |
|---|---|---|
| 택소노미 crosswalk + 크롤 마이닝 | ~$50~200 | 임베딩·집계, self-host |
| 멀티-LLM 앙상블 생성 (1만 카테고리 × 3 LLM × critic) | ~$300~3,000 | 1만×수회×수K토큰. 프리미엄 모델 써도 저렴 |
| **전문가 스키마 검수 (핵심 비용)** | **₩수천만** | 카테고리군별 도메인 전문가가 필수 Key·정의 검증 |
| **합계** | **≈ ₩3,000만~7,000만** | compute는 1%, **사람 검수가 지배** |

→ 1억 예산은 **전문가 검수 인력과 프리미엄 LLM 앙상블**에 쓰는 게 합리적이다. 1만 카테고리를 전수 전문가 검수하긴 어려우니, **상위 트래픽 카테고리(상위 ~1,000개가 GMV 대부분)는 전수 검수, 롱테일은 샘플 검수 + 자동 메트릭 게이트**로 차등 투입한다.

## 2.6 리스크와 완화

| 리스크 | 완화책 |
|---|---|
| 포괄성 미달(중요 속성 누락) | 멀티-LLM + critic + 경쟁사 합집합 + 검색로그 교차, golden 대비 coverage 게이트 |
| Key 난립(ME 위반) | 공유 라이브러리 강제, 동의어 클러스터링, 계층 구조화 |
| 카테고리 간 불일치 | 공유 속성 라이브러리에서 상속(같은 개념=같은 Key·단위) |
| 롱테일 카테고리 데이터 부족 | 택소노미 시드 + LLM 생성으로 cold-start, 부모 카테고리 속성 상속 |
| LLM이 없는 속성을 지어냄 | 실제 크롤 데이터에 근거, 전문가 검수, "근거 없는 항목" 표시 |
| 스키마 노후화(신상품 트렌드) | 주기적 재마이닝 + critic 재실행(→ 2-3 고도화 루프) |

> **연결:** 여기서 만든 스키마의 품질(특히 CE)을 *어떻게 측정·개선*하는지는 4장(2-3)에서 다룬다. 이 스키마를 바탕으로 *상품별 값을 추출*하는 파이프라인은 3장(2-2)이다.
