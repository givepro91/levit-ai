#1. 대규모 카테고리 분류 모델 설계

과제 [1] — 500만 상품을 1만 leaf 카테고리(계층 Tree)로 자동 분류. 입력: 상품명(텍스트) · 대표 이미지 · 상세페이지(텍스트/이미지)

#1.0 무엇이 진짜 어려운가

"1만 개 클래스 분류"라고 하면 보통 클래스가 많아서 어렵다고 생각한다. 그런데 진짜 난점은 클래스 수가 아니다.

  • 10K 클래스는 사실 "중간" 규모다. 진짜 XMC(extreme multi-label classification)는 수십만~수백만 클래스를 다룬다. 10K개에 대한 flat softmax 출력층은 10,000 × hidden(768) ≈ 7.7M 파라미터로 연산상 전혀 부담이 아니다. 따라서 "클래스가 많아서 flat이 불가능하다"는 흔한 논리는 이 문제에선 약하다.
  • 진짜 난점은 4가지다:
    1. 레이블 부재(label scarcity). "분류 모델을 새로 만든다" = 500만 건에 정답 leaf가 (충분히) 없다는 뜻이다. 학습 데이터를 어떻게 확보하느냐가 전체 설계를 좌우한다. (대부분의 설계가 이 질문을 건너뛴다 — 우리는 정면으로 다룬다. §1.3)
    2. 롱테일(long-tail). 1만 leaf 중 상위 ~100개가 데이터의 70~80%를 차지하는 파레토 분포가 일반적. 희소 클래스의 recall이 30% 밑으로 떨어지기 쉽다.
    3. 계층 일관성(hierarchical consistency). leaf만 맞히면 끝이 아니다. "가전 > 냉장고" 상품이 "패션 > 상의"로 가면 안 된다. 부모-자식 경로가 모순되지 않아야 한다.
    4. 멀티모달 노이즈. 이미지가 도움될 때(패션·인테리어)와 오히려 방해될 때(저화질·인포그래픽형 상세컷)가 갈린다.

이 4가지를 어떻게 푸느냐가 설계의 품질을 가른다.

#1.1.3 멀티모달 융합 — 텍스트 주도, 이미지는 tie-breaker

  • 일반적으로 텍스트가 이미지보다 분류에 강하고, 둘을 함께 쓰면 텍스트만 쓸 때보다 분명히 나아진다(향상폭은 카테고리·이미지 품질에 따라 다름).
  • 결정: 텍스트를 1차 신호, 이미지를 보조(tie-breaker)로 둔다. 이미지는 "빨간 원피스" vs "빨간 티셔츠"처럼 텍스트가 모호할 때 결정적이다. 단 이미지 품질 필터(해상도·블러·인포그래픽 감지)를 통과한 경우만 가중하고, 저품질이면 이미지 weight=0으로 fallback해 노이즈를 차단한다.
  • 비용 함의: 이미지 인코딩(ViT)은 임베딩 단계에서만 1회 — VLM 풀추론(건당 비싼)은 Stage 3 하드케이스에만. (전수 VLM이 예산을 깨는 이유 → §5)

#1.1.4 계층 일관성 — dynamic masking + hierarchical loss

  • Dynamic masking: cross-encoder가 leaf를 고를 때, 먼저 예측한 상위 레벨(대분류·중분류)의 유효 자식 집합으로 후보를 제한한다 → 부모-자식 모순 원천 차단.
  • Hierarchical loss: 학습 시 오답을 트리상 거리에 비례해 penalize한다. "냉장고→김치냉장고"(형제 오답)보다 "냉장고→원피스"(먼 오답)에 더 큰 손실. → 틀려도 "가까이" 틀리게 유도(다운스트림 피해 최소화).
  • Error cascade 완화: 상위 레벨 정확도를 별도 모니터링하고, 상위가 애매하면 곧장 Stage 3로 라우팅한다.

#1.1.5 보정(calibration)과 라우팅 임계값

  • Softmax 확률은 과신(overconfident) 경향 → temperature scaling 으로 보정, ECE(Expected Calibration Error) 를 검증셋으로 주기 측정.
  • 라우팅 임계값(0.85 / 0.55)은 고정값이 아니라 예산·정확도 트레이드오프의 다이얼이다. 임계값을 낮추면 자동 비율↑·비용↓·정확도↓. A/B로 캘리브레이션한다. (이 다이얼이 "1억 안에서 어디까지 품질을 살까"를 직접 조절하는 손잡이다.)

#1.2 정답 데이터를 어떻게 마련하나 — 이 설계의 관건

정답 leaf가 없는데 어떻게 cross-encoder를 학습시키나? 이 질문을 푸는 방식이 설계의 성패를 가른다.

3-소스 부트스트랩:

  1. 약한 신호 수확(weak labels). 크롤링 데이터에 판매자 지정 카테고리·태그·상세 스펙이 있으면 1차 시드로. 노이즈가 있지만 공짜이고 대량이다.
  2. LLM 교사 라벨링(핵심). 대표 표본 30만~50만 건을 상용 LLM(Solar Pro 2 / HyperCLOVA X / Gemini)으로 "이 상품의 leaf 카테고리는?"(top-5 후보 + 카테고리 정의를 in-context로 제공) 라벨링. 비용은 약 $70~200 에 불과(§5.2). 이게 distillation의 정답셋이 된다.
  3. 사람 golden set. 카테고리별 대표 상품을 전문가가 검수한 소량 고품질 셋 — 교사 라벨 품질 검증 + 최종 평가 기준.

→ LLM 교사가 만든 50만 라벨로 cross-encoder student를 distillation 하면, student는 교사 정확도의 85~95%를 추론 비용 1/50 이하로 달성한다. "비싼 LLM은 가르치는 데 한 번만 쓰고, 싼 student가 500만을 처리한다." — 이것이 예산을 지키며 정확도를 얻는 메커니즘이다.

산업 검증: Amazon은 fine-tuned 도메인 전문가가 top-K 후보를 제시하고 LLM이 그중에서 판별 하는 dual-expert 방식으로 높은 분류 정확도를 보고했고, Mercari는 라벨 없는 카탈로그를 OSS 임베딩 + LLM으로 일괄 재분류한 사례를 공유했다. 우리 설계는 이 두 검증된 패턴(전문가 후보 + LLM 판별, LLM 부트스트랩 재분류)을 합친 것이다.

#1.4 정확도 극대화 레버

  2. 계층 손실 — §1.1.4. 정확도 + 일관성 동시 개선.
  3. 롱테일 대비 contrastive learning — 희소 클래스를 anchor로 hard negative 구성, 임베딩 공간에서 tail 분리 강화.
  4. Active learning 루프 — 저신뢰·검수 케이스를 가장 모호한 클래스 경계에 집중 재학습.
  5. LLM self-summary few-shot — LLM이 각 카테고리의 "구분 기준"을 자기 생성 → reranker의 in-context 예시로 → 경계 카테고리 정밀도↑ (Amazon 패턴).
  6. 데이터 증강 — 상품명 paraphrase(back-translation/LLM), 이미지 augmentation으로 희소 클래스 보강.

#1.6 고도화 로드맵 (과제 1-2)

"1회성 전수 분류"로 끝이 아니라, 지속 운영되는 분류 시스템으로 키우는 단계별 계획.

#Phase 0 — 부트스트랩 & 전수 처리 (0~2개월) · 예산 1억의 대상

  • 약한 신호 + LLM 교사 라벨링 50만 → cross-encoder distillation → 초기 시스템.
  • 500만 전수 1회 분류 실행. 사람 검수로 golden set 1차 구축.
  • 게이트 지표: leaf P@1 ≥ 88%(자동확정 구간), 계층 일관성 ≥ 99%, 사람 큐 ≤ 5%.

#Phase 1 — 서빙 & 모니터링 (2~4개월)

  • 신규 상품 실시간 분류 API (p99 < 100ms). 신규는 Stage 1~2, 저신뢰만 Stage 3.
  • 운영 대시보드: 카테고리 분포 드리프트, 신뢰도 분포, 사람 큐 적체, 비용/건.
  • 게이트: 실시간 정확도 = 배치 정확도 ±2%p, 큐 처리 SLA 충족.

#Phase 2 — Active Learning & 롱테일 개선 (4~8개월)

  • 저신뢰·검수 결과를 학습셋에 환류 → 주기 재학습. 롱테일 집중 라벨링.
  • 보정 모니터링(ECE), 임계값 재튜닝.
  • 게이트: 롱테일(하위 50% 클래스) recall +10%p, ECE < 0.05.

#Phase 3 — 자기개선 & 다운스트림 연동 (8개월~)

  • 다운스트림 피드백을 라벨로: 검색 클릭·추천 전환·에이전트 정정 로그를 약한 라벨로 환류(공짜 라벨 공장).
  • taxonomy 진화(검색 기반이라 저비용), 교사 모델 주기 업그레이드 후 재distillation.
  • A/B로 라우팅 임계값·앙상블 가중 최적화.
  • 게이트: 분류 품질 → 검색/추천 지표(CTR·전환) 개선으로 연결 입증.

로드맵의 핵심: 예산 1억은 Phase 0(전수 처리)용이다. Phase 1+ 운영비는 별도지만, 설계가 처음부터 "싼 student가 대량 + 비싼 교사가 소량" 구조라 운영 한계비용이 낮게 유지된다 — 고도화의 핵심은 돈이 아니라 라벨 환류 루프다.

#2. 카테고리별 속성 스키마 구축 파이프라인

과제 [2-1] — 1만 카테고리 각각에 대해, 구매에 필요한 필수 속성(Key) 과 들어갈 수 있는 후보 값(Value) 집합 을 정의. 제약: 카테고리별 독립 스키마, MECE (특히 포괄성 CE — 소비자가 구매 시 고려하는 모든 핵심 정보를 커버).

#2.0 무엇이 어려운가 — 1만 개 스키마를 빠짐없이·겹치지 않게

예시(과제): 생수 카테고리 → Key: 수원지(지리산), 용량(500ml), 라벨 여부(무라벨).

카테고리마다 사전을 만드는 단순 작업처럼 보이지만, 진짜 난점은 셋이다.

  1. 규모 × 독립성. 1만 개 leaf 각각이 고유 스키마를 갖는다. 1만 번을 따로 만들면 (a) 비용·시간 폭증, (b) 일관성 붕괴("용량" vs "용량(ml)" vs "내용량"이 카테고리마다 제각각).
  2. 포괄성(CE) 증명의 어려움. "빠진 속성이 없다"를 어떻게 보장하나? 생성은 쉽지만 누락 없음을 증명하는 건 본질적으로 어렵다(열린 문제). 단일 LLM은 자기 사각지대를 모른다.
  3. 상호배타성(ME)·정규화. "수원지"와 "원수원지", "무라벨"과 "라벨없음"이 다른 Key/Value로 난립하면 안 된다.

핵심 통찰: 1만 개를 백지에서 독립 생성하지 않는다. 카테고리들은 속성을 대거 공유한다(모든 식품은 용량·유통기한, 모든 가전은 정격전압·크기). → 공유 속성 라이브러리(shared attribute library) + 카테고리별 오버레이 구조로 가면 비용·일관성·포괄성을 동시에 잡는다.

#2.2 Value 후보 집합과 타입 체계

각 Key는 타입을 갖고, 타입에 따라 Value 후보를 다르게 관리한다.

#2.3 필수(essential) vs 선택(optional) — 구매 관련성 랭킹

과제는 "필수 속성"을 요구한다. 모든 속성이 동급이 아니다.

  • 구매 관련성 점수(purchase-relevance) 로 Key를 랭킹: "이 속성을 모르면 구매 결정을 못 하는가?"
  • 신호: (a) LLM 판단(구매 가이드 기반), (b) 검색 로그(해당 속성으로 필터·검색하는 빈도), (c) PDP 노출 빈도(판매자가 항상 적는가).
  • 결과: 카테고리별 Key를 필수 / 권장 / 선택 3등급으로. 생수라면 필수=용량·종류, 권장=수원지·라벨여부, 선택=경도.
  • → 2-2 추출에서 필수 Key부터 추출해 비용을 우선순위화한다.

#2.4 MECE 운영화 — CE와 ME를 따로 공략

포괄성(CE, Collectively Exhaustive):

  1. 멀티-LLM 앙상블(다양성) + RAG(외부 지식) + 경쟁사 스키마 합집합(네이버·쿠팡·Amazon 동일 카테고리).
  2. Completeness critic 전용 패스(빠진 것 찾기).
  3. 소비자 구매여정 분석 프롬프트("구매 전 비교 시 확인하는 것?").
  4. 측정: 전문가 golden Key 셋 대비 coverage(recall) (→ 2-3).

상호배타성(ME, Mutually Exclusive):

  1. 임베딩+편집거리 동의어 클러스터링 → Key 병합.
  2. 계층 Key 구조: 겹치는 개념을 부모-자식으로(연결 > Wi-Fi/Bluetooth) → 중복 대신 구조.
  3. Key 정의서: 각 Key에 정의 + 예시 + 반례를 명문화(어노테이터·LLM 공용 가이드) → 경계 모호 제거.
  4. 측정: Key 쌍 임베딩 유사도 > 임계 → 중복 의심 자동 검출(→ 2-3).

#2.5 [2-1] 비용 — 여기선 compute가 아니라 "사람"이 비용

결정적 차이: 2-1은 카테고리 단위(1만) 작업이지 상품 단위(500만)가 아니다.

#3.0 무엇이 어려운가

[1]·[2-1]과 결이 다른 난점이 있다.

  1. 스키마 조건부(schema-conditioned). 추출할 Key가 상품마다 다르다 — 상품의 카테고리([1] 결과)가 어떤 스키마(2-1)를 적용할지 결정한다. "생수"엔 수원지를, "노트북"엔 CPU를 묻는다.
  2. 500만 × 멀티모달. 텍스트(상품명·상세)뿐 아니라 이미지에만 있는 속성(무라벨, 색상, 형태)이 있다. 그러나 이미지 추론(VLM)은 텍스트보다 비싸다 — 무차별 VLM은 비용을 깬다.
  3. 암묵적 속성(implicit). 본문에 "500ml"는 있어도 "용량 종류=소용량"은 추론해야 한다. 명시값 추출 + 추론 추출이 섞여 있다.
  4. 정규화(normalization)가 진짜 가치. "500미리", "0.5L", "500ML"가 모두 500ml로 통일돼야 검색·필터·추천이 작동한다. 추출보다 정규화가 다운스트림 가치를 좌우한다.
  5. "없음"의 3종 구분. 원문에 없음 vs 카테고리에 해당없음 vs 추출 실패 는 전혀 다르다. 뭉뚱그리면 데이터 신뢰가 깨진다.

#왜 텍스트 우선 2-pass인가 (비용 트레이드오프)

  • VLM은 텍스트 LLM보다 건당 ~7~70× 비싸다(§5). 500만 전수 VLM은 ~$350~수천이지만, 대다수 속성은 텍스트에 이미 있다.
  • 따라서 텍스트로 먼저 최대한 뽑고(거의 공짜), 이미지가 정말 필요한 잔여 필수 Key가 있는 상품에만 VLM을 건다 → VLM 호출량을 전수의 일부로 축소.
  • evidence_span(근거 구간)을 함께 뽑아 감사 가능성(auditability) 과 신뢰도 판단 근거를 남긴다.

#3.2 핵심 설계 포인트

Structured Output 강제. JSON Schema를 모델에 전달해 형식 위반·환각 Key를 차단. 출력 예:

{
  "용량":   {"value": "500ml", "confidence": 0.96, "evidence": "500ml 대용량", "source": "text"},
  "라벨여부": {"value": "무라벨", "confidence": 0.88, "evidence": "(이미지)",      "source": "image"},
  "수원지":  {"value": null,     "confidence": null, "reason": "미기재",           "source": "none"}
}

"없음" 3종 구분. reason ∈ {미기재(원문에 없음), 해당없음(카테고리 부적합), 추출실패(모델 불확실)}. 이 구분이 (a) 데이터 신뢰, (b) 어디를 보강할지(미기재→추가 크롤/추론, 추출실패→재시도/사람) 결정한다.

암묵적 속성. 명시 안 된 속성은 (a) VLM(이미지 추론), (b) LLM 사전지식·RAG(예: 브랜드→원산지)로 보강하되, 추론값은 source="inferred"로 명확히 태깅하고 신뢰도를 보수적으로. (명시값과 추론값을 섞지 않는다.)

정규화 = 다운스트림 가치. 단위 계산이 필요한 정규화는 LLM도 여전히 약한 영역이므로 규칙 우선 + 복잡 케이스만 LLM. 정규화한 값을 스키마의 표준 후보값에 맞춰, 검색·필터에 바로 쓰이는 구조값으로 만든다.

어려운 건 여러 모델로 교차검증. 확신이 낮거나 모순되거나 중요한 카테고리는 서로 다른 LLM 여러 개가 의견을 주고받아(debate) 합의로 견고함을 확보한다. 한 모델이 지어낸 값을 교차검증으로 걸러낸다.

#4. 속성 체계 평가 및 품질 관리

과제 [2-3] — 2-1에서 구축한 속성 체계(Key-Value 리스트)가 얼마나 정확하고 빠짐없이 구성됐는지 평가하고 고도화하는 방법(인간 피드백·자동 메트릭 등).

#4.0 무엇을 평가하는가 — 두 층위

품질은 두 곳에서 갈린다. 분리해서 측정한다.

  1. 스키마 품질(2-1) — Key 집합이 빠짐없고(CE), 겹치지 않고(ME), 맞는가(타입·정의).
  2. 추출 품질(2-2) — 그 스키마로 뽑은 값이 정확한가(P/R/F1), 정규화가 맞는가.

이 둘은 연결돼 있다. 스키마가 빈약하면(CE 미달) 추출이 아무리 정확해도 중요한 정보를 놓친다. 그래서 CE(포괄성) 평가가 가장 중요하고 가장 어렵다.

#포괄성(빠짐없음)은 어떻게 측정하나 — 어려운 문제와 네 가지 우회

"빠진 속성이 없다"는 모르는 것을 측정해야 하는 열린 문제다. 단일 지표로 못 풀고, 여러 약한 신호를 교차한다.

  1. Golden set recall — 전문가가 정의한 카테고리별 "필수 속성 정답"을 스키마가 얼마나 덮는가. (직접적이나 golden 자체가 완전하지 않을 수 있음)
  2. 검색 로그 gap 마이닝 (강력·실전적) — 사용자가 검색·필터에서 쓰는데 스키마에 없는 속성형 질의를 채굴한다. 예: "무라벨 생수"가 많이 검색되는데 스키마에 "라벨여부"가 없으면 → CE 갭이 실데이터로 드러난다. 다운스트림(검색)이 곧 포괄성 평가자.
  3. Held-out attribute 테스트 — 알려진 Key 하나를 일부러 빼고 파이프라인이 재발견하는지 검사 → 발견 메커니즘의 민감도 측정.
  4. 경쟁사 스키마 diff — 네이버·쿠팡·Amazon 동일 카테고리 속성 합집합 대비 누락 점검.

핵심: 검색 로그 gap 마이닝은 Levit/올웨이즈 같은 실서비스에서만 가능한 강점이다. "측정 못 하는 CE"를 "실사용자가 원하는데 못 채운 속성"이라는 측정 가능한 신호로 바꾼다.

#4.2 추출 품질 메트릭 (2-2)

  • Precision / Recall / F1 (micro-averaged) — golden 대비 값 정확도.
  • 값 정규화 정확도 — 정규화 후 값이 golden과 exact/semantic match 비율.
  • "없음" 분류 정확도 — 미기재/해당없음/추출실패 3종 구분이 맞는가(데이터 신뢰 직결).
  • 벤치마크 활용 — WDC-PAVE(정규화 포함), ImplicitAVE(암묵 속성), MAVE로 방법론 사전 검증.
  • 신뢰도 보정(calibration) — 모델 confidence와 실제 정확도가 일치하는가(ECE). 라우팅 임계값의 신뢰성 근거.

#4.3 Golden Set — 평가의 기준점

  • 카테고리별 대표 상품 100~200개 × 전체 Key를 전문가 2인 이상이 어노테이션, 불일치는 제3자 중재/다수결.
  • 용도: (a) 자동 메트릭의 정답, (b) 모델·프롬프트 선택 기준, (c) LLM-judge 캘리브레이션.
  • 차등 투자: 상위 트래픽 카테고리(~1,000개, GMV 대부분)는 풍부한 golden, 롱테일은 경량 golden + 자동 게이트. (1억 예산의 사람 비용이 주로 여기 — §4.6)

#4.5 LLM-as-Judge — 대량 평가, 단 편향 방어

전문가만으로 1만 카테고리·500만 상품을 평가할 수 없다. LLM-judge로 1차 스크리닝하되, 알려진 편향을 방어한다.

편향 방어
Verbosity(장문 선호) rubric 명시, 길이 무관 채점 기준
Position(순서 선호) swap consistency: 순서 바꿔 2회, 불일치 시 불신뢰
Self-enhancement(자기 선호) 이질 멀티-judge 앙상블(3종+ 다수결), 생성≠평가 모델 분리
전반 신뢰도 golden calibration: judge를 인간 golden에 보정, 편차 추적

운영 원칙: LLM 평가는 대량 1차 필터로만. 애매하거나 위험·고가치 카테고리는 사람에게 넘긴다. 자동 평가 결과는 사람 샘플로 주기적으로 다시 맞춘다.

#6. AI 활용 과정

과제 안내사항: "AI를 활용했다면, 과제 이해·문제 정의·아이디에이션·리서치 등 어떤 과정에서 어떻게 활용했는지 최대한 구체적으로 명시." 이 과제는 Claude(Opus 4.8) 기반 에이전트 워크플로우(Claude Code) 로 수행했다. AI를 "답을 대신 써주는 도구"가 아니라 검증된 보조 + 병렬 리서처 + 설계 파트너로 썼고, 사람(지원자)은 방향 설정·단계별 검토·최종 판단을 맡았다.

#6.1 원칙 — AI를 어떻게 신뢰했나

  1. 시간민감 사실은 기억이 아니라 출처. 모델 가격·버전·SOTA는 학습 cutoff 이후 바뀌므로, 전부 웹 1차 소스로 검증하고 URL·날짜를 부착했다. (그렇지 않으면 면접에서 무너진다.)
  2. 저작과 검증을 분리. 생성 패스와 검증 패스를 분리하고, 인용·수치는 빌드 전 adversarial 교차검증한다.
  3. 사람이 게이트. 요구사항 확정·포지셔닝·설계 방향·빌드 착수마다 사람 승인 게이트를 뒀다(자율 실행 ≠ 무단 진행).

  • 곧장 답을 쓰지 않고, 구조화된 인터뷰(6라운드) 로 요구사항을 결정화했다. 모호도를 0~1로 점수화해 100% → 13% 로 낮춘 뒤에야 작업을 시작.
  • 토폴로지 게이트로 작업을 3개 독립 컴포넌트([1]/[2]/산출물)로 잠그고, Contrarian 모드("트렌디함이 신뢰감을 깎지 않나?")·Simplifier 모드("과설계 아닌가?")로 가정을 흔들어 검증.
  • 산출: 합의된 스펙(포지셔닝=실용+트레이드오프 논증+비용감각/시니어 IC, 산출물=웹1차+PDF, 기술 stance=오픈소스 대량+상용 전략적+한국어 우선).

#② 아이디에이션

  • 포지셔닝과 "깔때기(funnel)" 설계 철학, 그리고 "비용은 binding 제약이 아니다" 라는 핵심 논점을 AI와의 대화로 도출. 회사 맥락(레브잇/올웨이즈, AI 쇼핑 에이전트)을 반영해 분류·속성 → 검색·추천·에이전트 → GMV 임팩트 서사를 구성.

#6.3 투명성 — 무엇이 검증됐고 무엇이 추정인가

  • 검증됨(1차 소스): 모델 가격(공식 pricing 직접 fetch), GPU 단가(RunPod·Lambda), 라이선스, 주요 산업사례(Amazon·Naver·Mercari).
  • 추정(가정 명시): 모든 비용 수치는 개략 추정이다. throughput·이미지 토큰은 ±30~50% 오차 가능. 단 결론(자릿수) 은 견고.
  • 검증 대상으로 남긴 것: 일부 최신 논문 인용은 출판 시점·ID를 빌드 단계에서 교차검증(검증 실패 시 강도 하향). 설계 자체는 검증된 표준 기법 기반이라 특정 인용의 진위와 무관하게 성립.

#6.4 메타 교훈

이 과제를 푼 방식 자체가 답의 일부다. deep-interview로 요구사항을 수렴 → 병렬 에이전트로 리서치를 검증·확장 → Opus로 설계를 합성 → 단일 소스로 산출물을 빌드 하는 흐름은, 그대로 AI/Data Product Engineer가 실무에서 LLM 파이프라인을 만드는 방식이다(요구 정의 → 검증된 리서치 → 깔때기 설계 → 평가 루프). 본문의 분류·속성 파이프라인이 "싼 모델 대량 + 비싼 모델 소수 + 사람 환류" 인 것과, 이 과제 수행이 "병렬 AI 대량 + 사람 게이트 검증" 인 것은 같은 원리다.

#7. 참고 문헌

아래 인용은 작성 중 독립 검증 패스(전용 에이전트가 각 논문/페이지의 제목·발표처·연도를 1차 소스로 대조)를 거쳤다. 검증 과정에서 발표처·수치 오류를 교정했고(예: Dual-Expert는 ACL이 아니라 EMNLP 2024, NGAME은 ICML이 아니라 WSDM 2023, ExtractGPT의 F1은 91%가 아니라 85%), 1차 소스로 확인되지 않은 수치는 본문에서 제거했다(예: 텍스트/이미지 단독 macro-F1 비교치). 본문 설계는 특정 인용의 진위와 무관하게 표준 기법에 기반한다.
